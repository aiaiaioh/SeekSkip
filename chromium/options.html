Query parameters — a field guide

What it is

The query parameter is the key in a search URL that carries your search terms: the q in google.com/search?q=cats. Seek Skip uses it in both directions — it reads the parameter to detect what you searched on the current engine, and writes it to build the same search on another engine. Everything after ? is a list of key=value pairs separated by &; only the one key you configure here matters to this extension.

How to find an engine's parameter

Search for something unmistakable, like TESTQUERY123, then look at the address bar and find which key= is holding it. That key is the parameter. If your search terms appear in the URL path instead (example.com/search/TESTQUERY123) rather than after a ?key=, see the limitations below.

Which URL to pair it with

Use the engine's results page URL with the query string stripped: search for something, then delete everything from ? onward. https://www.google.com/search, not https://www.google.com — the homepage often doesn't accept the parameter, or redirects. Exception: DuckDuckGo genuinely searches from its root (duckduckgo.com/?q=).

Behavior worth knowing

Encoding is automatic. Spaces, &, quotes, and non-ASCII characters are URL-encoded when building links and decoded when reading them — enter the parameter name only, never q= or a sample value.

Only the query hops. Extra parameters on the current page (language, region, safe-search, time filters) are engine-specific and are dropped when you switch. Your search terms carry over; your filters don't.

Detection is per-hostname. An engine is "recognized" when the page's hostname matches the engine URL's hostname (www. is ignored). Two engines on the same hostname with different parameters: only the first match wins.

Limitations

Path-based results pages — some sites accept a query parameter inbound but immediately rewrite the URL to a path or ID (Perplexity's /search/slug, Claude's /chat/uuid). These work as hop targets but can't be detected as the current engine, so no toolbar appears on them.

POST-based search forms — a few privacy frontends (e.g. some SearXNG instances in POST mode) send the query in the request body, not the URL. There's nothing in the URL to read or build, so those can't be supported either direction. Most have a GET mode you can enable.

Fragment queries — parameters after a # instead of ? aren't read. Rare in the wild.
